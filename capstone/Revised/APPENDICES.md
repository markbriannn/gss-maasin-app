## APPENDIX B: Interview Guide

**Target Respondents:** Clients, Service Providers, and Administrators in Maasin City, Southern Leyte

**Purpose:** To gather qualitative insights about user experiences, challenges, and perceptions regarding the General Service System.

### Interview Guide Questions

#### For Clients:

1. **Pre-System Experience:**
   - How did you usually find sideline service providers (carpenters, electricians, plumbers, etc.) before using the General Service System?
   - What challenges or difficulties did you experience when looking for reliable service providers?
   - How long did it typically take to find and hire a service provider using traditional methods?

2. **System Usage Experience:**
   - How did you learn about the General Service System?
   - What motivated you to try using the platform?
   - Can you describe your experience using the system for the first time?
   - Which features of the system do you find most helpful? (e.g., provider search, ratings, GPS tracking, payment)

3. **Impact and Benefits:**
   - In what ways has the General Service System improved the process of finding and hiring service providers?
   - How has the verification system (ID, clearances) affected your trust in hiring providers?
   - Has the system saved you time or money compared to traditional methods?

4. **Challenges and Suggestions:**
   - What difficulties have you encountered while using the system?
   - What improvements would you suggest to make the system better?
   - Are there any features you wish the system had?

#### For Service Providers:

1. **Pre-System Experience:**
   - How did you usually find clients for your sideline services before using the General Service System?
   - What challenges did you face in getting consistent work opportunities?
   - How did clients typically contact you and arrange for your services?

2. **Registration and Verification:**
   - Can you describe your experience during the registration and verification process?
   - What documents did you submit? (Government ID, Barangay Clearance, Police Clearance)
   - How long did it take for your account to be approved?

3. **System Usage and Impact:**
   - How has the system affected the number of job opportunities you receive?
   - What features do you find most useful? (e.g., job notifications, GPS tracking, payment system)
   - How has the online payment system (PayMongo/GCash) changed how you receive payments?

4. **Challenges and Suggestions:**
   - What difficulties have you experienced while using the platform?
   - How do you feel about the 5% platform service fee?
   - What improvements would make the system more beneficial for service providers?

#### For Administrators:

1. **System Management:**
   - Can you describe your role in managing the General Service System?
   - What are your main responsibilities? (Provider verification, booking approval, monitoring)
   - How many provider applications do you typically review per week?

2. **Verification Process:**
   - Can you walk through the process of verifying a new service provider?
   - What documents do you check? (Government ID, Barangay Clearance, Police Clearance, Profile Photo)
   - What criteria do you use to approve or reject provider applications?

3. **System Effectiveness:**
   - How effective is the admin dashboard for monitoring system activities?
   - What features of the admin panel do you find most useful? (Map view, analytics, provider management)
   - Have you encountered any issues or challenges in managing the platform?

4. **Observations and Suggestions:**
   - What improvements have you observed in service delivery since the system was implemented?
   - What challenges do users commonly report?
   - What enhancements would you recommend for the administrative functions?

### Interview Protocol:

- Obtain informed consent before starting the interview
- Explain the purpose of the research and how the data will be used
- Assure confidentiality and anonymity of responses
- Record interviews (with permission) or take detailed notes
- Allow participants to skip questions they're uncomfortable answering
- Thank participants for their time and insights

## APPENDIX D: Data Flow Diagrams (DFD)

This appendix contains the Data Flow Diagrams of the General Service System, illustrating how data moves between users, processes, and data stores throughout the system.

### D.1 Context Diagram

The Context Diagram shows the General Service System as a single process with external entities and major data flows:

**External Entities:**
- Client
- Service Provider
- Administrator
- PayMongo Payment Gateway
- Brevo Email Service
- Semaphore SMS Service

**Major Data Flows:**
- User registration and authentication data
- Service requests and bookings
- Payment transactions
- Verification documents
- Notifications and messages
- GPS location data
- Feedback and ratings

### D.2 Level 1 Data Flow Diagram

The Level 1 DFD decomposes the system into major processes:

**Process 1.0: Client Registration**
- Input: Personal info, email, phone, password, profile photo
- Output: Client account, verification codes
- Data Stores: Users Database

**Process 2.0: Provider Registration**
- Input: Personal info, services, documents (ID, clearances), profile photo
- Output: Provider account (pending status)
- Data Stores: Users Database, Documents Storage

**Process 3.0: Admin Verification of Provider**
- Input: Provider application, submitted documents
- Output: Approval/rejection decision, verified provider account
- Data Stores: Users Database

**Process 4.0: Service Discovery and Booking Creation**
- Input: Service requirements, schedule, location, photos
- Output: Booking request (pending admin approval)
- Data Stores: Bookings Database

**Process 5.0: Admin Review Booking**
- Input: Booking request details
- Output: Approved/rejected booking
- Data Stores: Bookings Database

**Process 6.0: Provider Response to Job**
- Input: Job notification, provider decision
- Output: Accepted/declined job status
- Data Stores: Bookings Database

**Process 7.0: Payment Processing**
- Input: Payment method, amount, booking details
- Output: Payment confirmation, transaction record
- Data Stores: Transactions Database, PayMongo Gateway

**Process 8.0: Service Execution and Tracking**
- Input: Job status updates, GPS location
- Output: Real-time tracking data, completion status
- Data Stores: Bookings Database, Location Data

**Process 9.0: Feedback and Rating**
- Input: Rating score, review text
- Output: Updated provider rating, review record
- Data Stores: Reviews Database, Users Database

**Process 10.0: Messaging and Communication**
- Input: Messages between users
- Output: Conversation threads
- Data Stores: Messages Database

**Process 11.0: Notification Management**
- Input: System events, user actions
- Output: Push notifications, SMS, emails
- Data Stores: Notifications Database

**Process 12.0: Gamification Management**
- Input: User activities, completed bookings
- Output: Points, tiers, badges, leaderboard rankings
- Data Stores: Gamification Database

**Data Stores:**
- D1: Users (Clients, Providers, Admins)
- D2: Bookings
- D3: Transactions
- D4: Reviews
- D5: Messages
- D6: Notifications
- D7: Documents

### E.2 Relationships

- USERS (1) ←→ (1) CLIENTS (Inheritance/IS-A)
- USERS (1) ←→ (1) PROVIDERS (Inheritance/IS-A)
- USERS (1) ←→ (1) ADMINISTRATORS (Inheritance/IS-A)
- CLIENTS (1) ←→ (M) BOOKINGS (One client can have many bookings)
- PROVIDERS (1) ←→ (M) BOOKINGS (One provider can have many bookings)
- BOOKINGS (1) ←→ (1) TRANSACTIONS (One booking has one transaction)
- BOOKINGS (1) ←→ (1) REVIEWS (One booking can have one review)
- PROVIDERS (1) ←→ (M) REVIEWS (One provider can have many reviews)
- USERS (1) ←→ (M) MESSAGES (One user can send many messages)
- USERS (1) ←→ (M) NOTIFICATIONS (One user can have many notifications)
- USERS (M) ←→ (M) CONVERSATIONS (Many-to-many through participants)

### E.3 Cardinality and Constraints

- Each BOOKING must have exactly one CLIENT and one PROVIDER
- Each TRANSACTION must be linked to exactly one BOOKING
- Each REVIEW must be linked to exactly one BOOKING and one PROVIDER
- PROVIDERS must have verificationStatus = 'approved' to receive jobs
- BOOKINGS must have status = 'completed' before REVIEW can be created
- TRANSACTIONS are created when BOOKING status changes to 'completed'

## APPENDIX H: ISO/IEC 9126 Evaluation Instrument

This appendix contains the official evaluation instrument used to assess the General Service System based on ISO/IEC 9126 software quality criteria.

### H.1 ISO/IEC 9126 Quality Characteristics

The ISO/IEC 9126 standard defines six quality characteristics for software evaluation:

1. **Functionality** - The capability of the software to provide functions that meet stated and implied needs
2. **Reliability** - The capability to maintain a specified level of performance under stated conditions
3. **Usability** - The effort needed for use and the individual assessment of such use
4. **Efficiency** - The relationship between the level of performance and the amount of resources used
5. **Maintainability** - The effort needed to make specified modifications
6. **Portability** - The ability to be transferred from one environment to another

### H.2 Evaluation Criteria and Metrics

#### Functionality Metrics

**Sub-characteristics:**
- **Suitability** - Does the system provide appropriate functions for specified tasks?
- **Accuracy** - Does the system provide correct or agreed results?
- **Interoperability** - Can the system interact with specified systems?
- **Security** - Does the system protect information and data?
- **Compliance** - Does the system adhere to standards and regulations?

**Evaluation Questions:**
1. The system provides all necessary features for service booking and management
2. Registration and verification processes work correctly
3. Payment processing produces accurate transaction records
4. The system integrates properly with external services (PayMongo, Brevo, Semaphore)
5. User data is protected through authentication and encryption
6. The system complies with data privacy regulations

**Rating Scale:** 1 (Strongly Disagree) to 5 (Strongly Agree)

#### Reliability Metrics

**Sub-characteristics:**
- **Maturity** - How often does the system fail?
- **Fault Tolerance** - Can the system maintain performance despite faults?
- **Recoverability** - Can the system recover data after failure?

**Evaluation Questions:**
1. The system performs consistently without crashes or errors
2. Data entered is saved correctly and not lost
3. The system handles errors gracefully without complete failure
4. The system recovers quickly from temporary issues
5. User sessions are maintained reliably

**Rating Scale:** 1 (Strongly Disagree) to 5 (Strongly Agree)

#### Usability Metrics

**Sub-characteristics:**
- **Understandability** - Can users understand the system's purpose and functions?
- **Learnability** - How easy is it to learn to use the system?
- **Operability** - How easy is it to operate and control the system?
- **Attractiveness** - Is the interface appealing to users?

**Evaluation Questions:**
1. The system is easy to learn and navigate
2. Instructions and labels are clear and understandable
3. The interface is user-friendly and intuitive
4. Error messages are helpful and guide users to solutions
5. The system design is visually appealing
6. Users can accomplish tasks efficiently

**Rating Scale:** 1 (Strongly Disagree) to 5 (Strongly Agree)

#### Efficiency Metrics

**Sub-characteristics:**
- **Time Behavior** - How fast does the system respond?
- **Resource Utilization** - How efficiently does the system use resources?

**Evaluation Questions:**
1. Pages and features load quickly
2. The system responds promptly to user actions
3. Search and filter functions work efficiently
4. The system performs well even with multiple users
5. The system saves time compared to traditional methods

**Rating Scale:** 1 (Strongly Disagree) to 5 (Strongly Agree)

#### Maintainability Metrics (For Developers/Technical Evaluators)

**Sub-characteristics:**
- **Analyzability** - How easy is it to diagnose deficiencies?
- **Changeability** - How easy is it to modify the system?
- **Stability** - How likely are unexpected effects from modifications?
- **Testability** - How easy is it to validate modifications?

**Evaluation Questions:**
1. Code is well-organized and documented
2. System architecture is modular and maintainable
3. Changes can be implemented without affecting other components
4. The system can be tested effectively
5. Debugging and troubleshooting are straightforward

**Rating Scale:** 1 (Strongly Disagree) to 5 (Strongly Agree)

#### Portability Metrics

**Sub-characteristics:**
- **Adaptability** - Can the system adapt to different environments?
- **Installability** - How easy is it to install the system?
- **Replaceability** - Can the system replace another system?
- **Co-existence** - Can the system co-exist with other systems?

**Evaluation Questions:**
1. The system works on different devices (desktop, mobile, tablet)
2. The system works on different browsers (Chrome, Firefox, Safari)
3. The web and mobile versions provide consistent functionality
4. The system can be accessed without complex installation
5. The system adapts to different screen sizes

**Rating Scale:** 1 (Strongly Disagree) to 5 (Strongly Agree)

### H.3 Evaluation Scoring Guide

**Interpretation of Mean Scores:**

| Mean Score Range | Verbal Interpretation | Description |
|------------------|----------------------|-------------|
| 4.50 - 5.00 | Excellent | The system excellently meets the quality criterion |
| 3.50 - 4.49 | Very Good | The system meets the quality criterion very well |
| 2.50 - 3.49 | Good | The system adequately meets the quality criterion |
| 1.50 - 2.49 | Fair | The system partially meets the quality criterion |
| 1.00 - 1.49 | Poor | The system does not meet the quality criterion |

### H.4 Respondent Profile for Evaluation

**Target Respondents:**
- Clients who have used the system (minimum 3 bookings)
- Service Providers who have completed jobs (minimum 3 jobs)
- System Administrators
- Technical evaluators (IT professionals, developers)

**Minimum Sample Size:** 30 respondents (10 clients, 10 providers, 5 admins, 5 technical evaluators)

### H.5 Data Collection and Analysis

**Data Collection Method:**
- Online survey questionnaire
- Face-to-face interviews with structured questions
- System usage observation

**Statistical Analysis:**
- Calculate mean scores for each quality characteristic
- Calculate overall system quality score
- Perform frequency distribution analysis
- Compare scores across different user groups

**Validity and Reliability:**
- Content validity established through expert review
- Reliability tested using Cronbach's Alpha (target: α ≥ 0.70)
- Pilot testing conducted with 10 respondents before full deployment
